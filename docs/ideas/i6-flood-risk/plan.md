Роль в общем плане (см. [docs/plan/development-plan.md](../../plan/development-plan.md)): **основной кейс**. Гейт 12.07 18:00 — есть ли таргет (разметка затоплений-2024). Нет → переключение на И-9.

## Цель MVP (к 17.07)

Для одного региона (СКО или Актюбинская обл.): скоринг риска весеннего затопления по населённым пунктам + объяснение каждого скора + очередь превентивных мер, утверждаемая человеком. Живое демо в существующем приложении.

## План работ

### Данные (B, 11–12.07)
| Датасет | Источник | Назначение | Риск доступа |
| --- | --- | --- | --- |
| Список НП региона (КАТО, координаты, население) | data.egov.kz / OSM / GeoNames | объекты скоринга | низкий |
| Затопления-2024 (таргет) | Sentinel-1 SAR апрель-2024 (Copernicus Browser / AWS Open Data) + сводки ДЧС и СМИ | обучение модели | **средний — гейт 12.07** |

| Землепользование | ESA WorldCover | признаки | низкий |

### ML (B, 12–14.07)
1. Разметка: маска воды по Sentinel-1 (порог VV) на пике паводка → пересечение с буфером НП → таргет `затоплен/нет`; сверка со сводками ДЧС.
2. Датасет признаков на НП (~200–600 строк): высота над ближайшей рекой, уклон, расстояние до реки, площадь водосбора выше, снегозапас в % от нормы, сумма осадков марта, скорость перехода через 0 °C, доля поймы в буфере, население.
3. Модель: LightGBM (бинарная классификация → калиброванная вероятность → скор 0–100). Baseline для сравнения: логистическая регрессия и «эвристика по высоте над рекой».
4. Валидация: stratified CV + hold-out по районам; метрики ROC-AUC и precision@20 (в терминах пользователя: «из топ-20 сёл по скору сколько реально топило»).
5. SHAP per-НП → экспорт top-5 факторов в БД.

### Приложение (A, 11–15.07)
1. Таблица `Settlements` (+метрики на НП) + миграция; API `GET /api/settlements?module=flood-risk`.
2. Карта: точки НП поверх областей, цвет по скору; клик → деталка.
3. Деталка НП: скор, панель «Почему» (SHAP-факторы человеческим языком), очередь мер.
4. Очередь превентивных мер: генерация черновика по правилам (скор × население × уязвимые объекты), статусы «предложено → утверждено комиссией / отклонено» — human-in-the-loop с аудиторским следом.
5. Демо-путь: логин → карта региона → топ-20 НП → «почему» → утверждение меры.

### Требования и запреты ТЗ

- ✅ Одна конкретная проблема; понятный пользователь (комиссия ДЧС / отдел ЧС акимата)
- ✅ Почему не обычная автоматизация: карта без модели не отвечает «какое село затопит при текущем снегозапасе»
- ✅ AI/Data-driven компонент из списка §6: прогнозирование риска + скоринг и ранжирование + анализ изображений (SAR)
- ✅ Explainability §7: факторы, данные, ограничения, финальное решение — человек
- ✅ Роль человека §8: комиссия утверждает меры, система только предлагает
- ✅ Не повторяет Decentrathon 5.0; не «портал всего»; не чёрный ящик
- ⚠️ §5 «работает на реальных данных»: признаки реальные, таргет — собственная разметка; в доках честно описать методику и ограничения (это разрешено: «данные, собранные командой самостоятельно», §9)

### Главный риск идеи

Таргет. Если SAR-разметка к 12.07 не даёт ≥30–40 достоверно затопленных НП — модель не обучить, переключаемся на И-9 без потери наработок приложения.
