# Зимняя обстановка (winter-risk)

Индекс зимней опасности **0–100 по 16 областям** Казахстана за сезоны **2020–2026**,
из четырёх погодных под-индексов. Реализовано по образцу паводка: данные лежат в
репозитории и сидятся в БД пакетом на старте, фронт только отображает.

## Данные

Open-Meteo **Archive (ERA5)**, без ключа. На область берётся суточная погода за сезон
(ноя(Y−1)…мар(Y)) + пиковая высота снежного покрова (`snow_depth`, почасовой).

## Индекс и под-индексы

Каждый под-индекс 0..1, итог — взвешенный композит (строго 0–100):

```
risk = 100 · (0.30·гололёд + 0.30·метель + 0.20·снегонагрузка + 0.20·холод)
```

- **Гололёд** — дни ледяного дождя (дождь при tmax ≤ 1 °C) + переходы через 0 °C (наледь). Пороги с запасом, не насыщается.
- **Метель** — дни с ветром ≥ 11 м/с при снегопаде.
- **Снегонагрузка** — по **пиковому снежному покрову** (snow_depth), а не по сумме снега.
- **Холод** — дни ≤ −25 °C + насколько минимум ниже −25 (индекс жёсткости).

Формулы и пороги — `scripts/winter_fetch.py::sub_indices`. Все под-индексы тоже 0–100
и показываются в карточке как разбор «почему».

## Пайплайн (как у паводка)

1. **Разово, с интернетом:** `python scripts/winter_fetch.py [2020] [2026]` →
   `data/processed/winter_regions.csv` (16 обл. × 7 сезонов = 112 строк; колонки
   `iso, season, risk_score, idx_glaze, idx_blizzard, idx_snowload, idx_cold`).
   Один запрос на сезон (быстро), устойчив к обрывам, продолжает с места остановки.
2. **`backend/Dockerfile`** запекает CSV в образ (`seed-data/`), как скоры паводка.
3. **`DataFileSeeder`** при пустой БД грузит его пакетом в `RegionMetric`
   (`Module=winter-risk`, 5 метрик × область × сезон). Без внешних HTTP-запросов.
4. **`frontend/src/views/ideas/WinterRisk.vue`** читает `/api/regions/metrics/winter-risk`
   (`risk_score` + `idx_*` по выбранному сезону) и отображает.

## Фронт

Карта `KazakhstanMap` с синей палитрой (`palette`) и фиксированной шкалой 0–100
(`domain-min/domain-max`). Клик по области → индекс и разбор по 4 под-индексам.
Пункт меню — «Зимняя обстановка» (роут `/risks/winter`).

## Обновить или добавить сезоны

```
python scripts/winter_fetch.py 2020 2026      # перезаписывает/дополняет CSV
docker compose down -v && docker compose up --build   # пересев БД
```

## Расширение на ML (следующий шаг)

Сейчас `risk_score` — прозрачная формула (baseline, роль как у FWI-индекса пожара).
Чтобы сделать ML как у паводка, менять нужно **только расчёт**, не формат и не фронт:

- **признаки** — те же 4 под-индекса (или их сырые входы) по области×сезону;
- **таргет** — реальные зимние ЧП (погодные ДТП / перекрытия трасс / коммунальные
  аварии), из официальных источников: `data.egov.kz`, КПСиСУ `qamqor.gov.kz`, КазАвтоЖол;
- **модель** — LightGBM + SHAP → калиброванный `risk_score` и факторы со знаком;
- **точка расширения** — `sub_indices()` / итоговый `risk_score` в `scripts/winter_fetch.py`.
  CSV-формат, сидер и фронт остаются без изменений — модель просто заменяет расчёт числа.
