# Превентивное управление природными рисками

**GovTech Camp — команда IcyBeard.** Система поддержки решений для ДЧС и акиматов: риск-скоринг населённых пунктов, объяснение каждого вывода и очередь превентивных мер с утверждением человеком.

## Проблема

Государство управляет природными ЧС **реактивно**: паводок-2024 — >118 тыс. эвакуированных и ~₸400 млрд ущерба, при этом превентивные меры (обследование дамб, очистка русел, планы эвакуации) планируются субъективно, «по прошлому опыту», а заявки акиматов на превенцию не имеют доказательной базы. Обычной автоматизацией это не решается: свести данные на одну карту можно, но ответить «какое село затопит при текущем снегозапасе и куда направить ограниченный бюджет в первую очередь» может только модель, обученная на реальных событиях.

## Пользователь и сценарий

Комиссия ДЧС / отдел ЧС акимата области: открывает карту риска → видит ранжированные НП и «почему» по каждому → система предлагает черновики превентивных мер (приоритет = риск × население) → комиссия утверждает или отклоняет, каждое решение фиксируется (кто/когда). AI не принимает финальных решений.

## Решение — одна платформа, сезонный цикл рисков

| Контур | Что делает | Статус |
| --- | --- | --- |
| **Паводковый риск-скоринг** (основной кейс, пилот — СКО) | LightGBM на фактах затоплений-2024: скор 0–100 для 520 НП, TreeSHAP-объяснения в UI, ретроспектива сезонов 2010–2026, очередь мер | ✅ работает |
| **Пожарная live-обстановка** | Метео-индекс по 174 районам (Open-Meteo, сейчас), очаги NASA FIRMS за 24 ч, анимация ветра, слои NASA GIBS, автообновление 15 мин | ✅ работает |
| **Зимняя обстановка** | Композитный индекс зимней опасности по областям (гололёд + метель + снегонагрузка + холод, ERA5-архив), сезоны 2020–2026, разбор по под-индексам | ✅ работает |

Паводки — весной, пожары — летом, гололёд и метели — зимой: один движок (карта → риск → объяснение → действие) закрывает круглогодичный цикл природных рисков. Это не «портал всего», а один паттерн превентивного управления на сезонных рисках одного домена.

## AI/ML-подход и честные метрики

- **Таргет:** факт затопления НП весной 2024 (разметка из сводок МЧС/СМИ с источником на каждую запись; методика и PU-ограничения — [docs/data/flood-labels-methodology.md](docs/data/flood-labels-methodology.md)).
- **Признаки:** рельеф (Copernicus DEM: превышение над рекой, расстояние до реки), гидрография (OSM), снегозапас и осадки в % от нормы (ERA5-Land, 2000–2026).
- **Модель паводков:** LightGBM; **ROC-AUC 0.867** (честная 5-fold out-of-fold оценка) против **0.794** у прозрачного baseline-индекса; **precision@20 = 0.40** — 8 из топ-20 сёл по скору реально топило (базовая частота 3.3%).
- **Модель пожаров (район×день, 186 тыс. строк):** LightGBM на архиве FIRMS 2021–2025 + погоде; тест — весь 2025: **AUC 0.832 против 0.686** у композита, capture@top-20% **55% против 36%** — патрули по модели накрывают в 1.5 раза больше реальных пожарных дней ([ml/i9-fire-risk/](ml/i9-fire-risk/)); на карте — режим «ML-прогноз на сегодня».
- **Explainability:** встроенный TreeSHAP — вклад каждого фактора виден в карточке НП («Река в 0.5 км: +1.2», «Снегозапас 142% нормы: +0.8»); модель объясняет и нули («Альжан: превышение 58 м → риск 0 — меры не требуются»).

## Быстрый старт (одна команда, нужен только Docker)

Работает одинаково на Windows / macOS / Linux — `make` не требуется:

```bash
docker compose up --build -d
```

Весь стек в docker-контейнерах: PostgreSQL (+PostGIS) → API → веб-клиент (nginx). Миграции, справочники и **данные пилота (520 НП СКО, ML-скоры, 17 сезонов) загружаются автоматически** при первом старте — ничего доустанавливать и настраивать не нужно (`.env` опционален: у compose есть значения по умолчанию).

- **Веб-клиент:** http://localhost:5173 — вход `admin@icybeard.local` / `ChangeMe123!`
- **API:** http://localhost:5080 (`/api/health`); из веб-клиента `/api` проксирует nginx
- Остановить: `docker compose down`; полный сброс БД: `docker compose down -v`
- На macOS/Linux есть сокращения: `make up` / `make upd` / `make down` / `make clean` / `make logs`

**Опционально:** очаги NASA FIRMS на карте пожарной обстановки требуют бесплатный ключ (2 минуты: firms.modaps.eosdis.nasa.gov/api → Generate MAP_KEY → в `.env`: `FIRMS_MAP_KEY=...`). Без ключа всё остальное работает, страница честно сообщает, чего не хватает.

**Демо-сценарий (2–3 минуты):** войти → Дашборд → «Паводковый риск · СКО»: карта, клик по селу → скор и факторы «почему» → «Сгенерировать черновики мер» → утвердить меру (фиксируется кто/когда) → селект «Сезон» и график снегозапаса → «Пожарная обстановка · live»: очаги, ветер, метео-индекс по районам → «Зимняя обстановка»: индекс по областям с разбором (гололёд/метель/снегонагрузка/холод).

Для локальной разработки с hot-reload (нужны .NET SDK 10 и Node.js 20+): `make db` + `make backend` + `make frontend`. Детали — в [backend/README.md](backend/README.md) и [frontend/README.md](frontend/README.md).

## Стек

- **Frontend:** Vue 3 + PrimeVue (шаблон [Sakai](https://sakai.primevue.org/)); карта — Leaflet + анимация ветра (leaflet-velocity) + спутниковые слои NASA GIBS
- **Backend:** .NET 10 minimal API, JWT-аутентификация, роли Admin/Analyst, прокси NASA FIRMS
- **DB:** PostgreSQL + PostGIS, EF Core миграции + автосид данных пилота при первом старте
- **ML/Data:** Python-пайплайн в `ml/` и `scripts/` (LightGBM + TreeSHAP); живая погода — Open-Meteo без ключей

## Как данные попадают на карту

1. Скрипты (`scripts/`) собирают признаки: OSM (НП, реки), Copernicus DEM (рельеф), ERA5-Land (снегозапас/осадки, 2000–2026).
2. `ml/i6-flood-risk/train.py` обучает LightGBM на разметке-2024 и отдаёт скоры + TreeSHAP-факторы; `score_years.py` скорит каждый сезон.
3. Загрузка: `PUT /api/settlements/metrics` (роль Admin) — либо автосид из CSV при первом старте контейнера.
4. Фронтенд: `GET /api/settlements/metrics/flood-risk?metricKey=risk_score&period=YYYY` → точки на карте, факторы в панели «Почему».
5. Live-слои мимо БД: Open-Meteo (погода/ветер), NASA FIRMS через `/api/fire/hotspots`, тайлы NASA GIBS.

## Правила для команды (2 человека)

- Документы кладём **только в `docs/`**, в подходящую тематическую подпапку.
- Внутренние коды модулей в путях/данных: `flood-risk`, `fire-risk` (наружу — «платформа природных рисков»).
- Сырые датасеты — в `data/raw/` (не коммитим, в `docs/data/` описываем, откуда скачать).
- Работаем в ветках `feat/<модуль>-<что-делаем>`, мерджим в `main` через PR.

## Известные ограничения

- Границы (`frontend/public/geo/kz-regions.geojson` и `kz-districts.geojson`) — geoBoundaries 2017 г.: 16 областей (без Абайской, Жетысу, Улытау) и 174 района. Заменить синхронно с сидером регионов (`DbSeeder.cs`).
- Разметка затоплений-2024 — 17 позитивов из СМИ (PU-постановка, методика в `docs/data/flood-labels-methodology.md`); модель обучена на одном событии.
- Очаги FIRMS берутся по прямоугольнику — видны точки и за границей РК.
- Метео-индекс пожаров — прозрачный baseline (не ML); модель по ячейкам — план основной программы.
