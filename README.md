# Превентивное управление природными рисками

**GovTech Camp — команда IcyBeard.** Система поддержки решений для ДЧС и акиматов: риск-скоринг населённых пунктов, объяснение каждого вывода и очередь превентивных мер с утверждением человеком.

> **🌐 Живое демо (без установки): https://dc.jurek.kz** — вход `admin@icybeard.local` / `ChangeMe123!`. Тот же код и данные, что поднимает `docker compose up --build` (см. [Быстрый старт](#быстрый-старт-одна-команда-нужен-только-docker)).

## Проблема

Государство управляет природными ЧС **реактивно**: паводок-2024 — >118 тыс. эвакуированных и ~₸400 млрд ущерба, при этом превентивные меры (обследование дамб, очистка русел, планы эвакуации) планируются субъективно, «по прошлому опыту», а заявки акиматов на превенцию не имеют доказательной базы. Обычной автоматизацией это не решается: свести данные на одну карту можно, но ответить «какое село затопит при текущем снегозапасе и куда направить ограниченный бюджет в первую очередь» может только модель, обученная на реальных событиях.

## Пользователь и сценарий

Комиссия ДЧС / отдел ЧС акимата области: открывает карту риска → видит ранжированные НП и «почему» по каждому → система предлагает черновики превентивных мер (приоритет = риск × население) → комиссия утверждает или отклоняет, каждое решение фиксируется (кто/когда). AI не принимает финальных решений.

**Измеримый результат:**

- приоритизация превенции вместо «по опыту»: в топ-20 НП по скору модель ловит 8 реально затопленных сёл против ~0–1 при случайном выборе (precision@20 = 0.40 при базовой частоте 3.3%);
- патрулирование по ML-прогнозу пожаров накрывает **в 1.5 раза больше** реальных пожарных дней при том же ресурсе (capture@top-20%: 55% против 36% у эвристики);
- заявка акимата на превентивные меры получает доказательную базу: скор + факторы + правило — вместо субъективного обоснования;
- аудиторский след решений (кто/когда утвердил каждую меру) — прозрачность для проверок и анализа «почему не сделали».

## Решение — одна платформа, сезонный цикл рисков

| Контур | Что делает | Статус |
| --- | --- | --- |
| **Паводковый риск-скоринг** (основной кейс, пилот — СКО) | LightGBM на фактах затоплений-2024: скор 0–100 для 520 НП, TreeSHAP-объяснения в UI, ретроспектива сезонов 2010–2026, очередь мер | ✅ работает |
| **Пожарная live-обстановка** | Риск по 174 районам = live-метео (Open-Meteo) + историческая частота очагов FIRMS; режим «ML-прогноз на сегодня» (дневная LightGBM); очаги за 24 ч, анимация ветра, слои NASA GIBS, обновление раз в час; очередь превентивных мер по районам | ✅ работает |
| **Зимняя обстановка** | Композитный индекс зимней опасности по 174 районам (гололёд + метель + снегонагрузка + холод, ERA5-архив), сезоны 2020–2026, разбор по под-индексам; режим «ML-прогноз» — LightGBM на предупреждениях Казгидромета: ретроспектива сезонов с факторами «почему» и прогноз на сегодня; очередь превентивных мер по районам | ✅ работает |
| **Страница «Данные»** | Загрузка датасетов (Admin): скоры паводков (CSV) ингестируются сразу — карта сезона пересчитывается; остальные наборы — реестр для офлайн-пайплайна ML | ✅ работает |

Паводки — весной, пожары — летом, гололёд и метели — зимой: один движок (карта → риск → объяснение → действие) закрывает круглогодичный цикл природных рисков. Это не «портал всего», а один паттерн превентивного управления на сезонных рисках одного домена. Кейс новый для команды — не повторяет задачи Decentrathon 5.0.

**Две шкалы в интерфейсе: риск-скор 0–100 и «приоритет» до ~550.** Риск-скор везде в шкале 0–100 — карта, бейджи, гистограммы распределения, пороги генерации мер. «Приоритет» в очереди мер — другая величина: порядок рассмотрения для комиссии. У паводков приоритет = скор × lg(население), поэтому его максимум ≈ 550 (Петропавловск: скор 99.9 × lg(222 703 чел.) = 534.2) — так город со скором 70 встаёт в очереди выше разъезда со скором 90, где цена ошибки несопоставима. У пожаров и зимы меры районные, население не взвешивается — там приоритет равен скору района (≤ 100). На дашборде, где очередь общая для всех контуров, приоритеты нормируются к максимуму своего контура — иначе паводковые меры всегда вытесняли бы районные. Формула приоритета показывается в диалоге «Почему рекомендовано» у каждой меры.

**Режимы карты (единая логика во всех контурах).** «Индекс» — прозрачная объяснимая формула-baseline (у пожаров: метео сейчас + история очагов FIRMS; у зимы: гололёд + метель + снегонагрузка + холод из ERA5). «ML-прогноз» — обученная LightGBM-модель с SHAP-факторами «почему» в карточке объекта (паводки — скоринг НП, пожары — прогноз на сегодня, зима — ретроспектива сезонов и прогноз на сегодня в зимние месяцы). Модель и формула всегда показываются рядом — видно, где ML реально выигрывает (метрики против baseline — в `ml/*/metrics.json` и `ml/README.md`). Блок «История» в карточке объекта — фактические данные прошлых лет (очаги FIRMS у пожаров, индексы прошлых зим у зимы); у паводков ту же роль играет ретроспектива сезонов (пикер сезона + график снегозапаса, модель скорит погоду каждого года).

## AI/ML-подход и честные метрики

- **Таргет:** факт затопления НП весной 2024 (разметка из сводок МЧС/СМИ с источником на каждую запись; методика и PU-ограничения — [docs/data/flood-labels-methodology.md](docs/data/flood-labels-methodology.md)).
- **Признаки:** рельеф (Copernicus DEM: превышение над рекой, расстояние до реки), гидрография (OSM), снегозапас и осадки в % от нормы (ERA5-Land, 2000–2026).
- **Модель паводков:** LightGBM; **ROC-AUC 0.867** (честная 5-fold out-of-fold оценка) против **0.794** у прозрачного baseline-индекса; **precision@20 = 0.40** — 8 из топ-20 сёл по скору реально топило (базовая частота 3.3%).
- **Модель пожаров (район×день, 186 тыс. строк):** LightGBM на архиве FIRMS 2021–2025 + погоде; тест — весь 2025: **AUC 0.832 против 0.686** у композита, capture@top-20% **55% против 36%** — патрули по модели накрывают в 1.5 раза больше реальных пожарных дней ([ml/i9-fire-risk/](ml/i9-fire-risk/)); на карте — режим «ML-прогноз на сегодня».
- **Explainability:** встроенный TreeSHAP — вклад каждого фактора виден в карточке НП («Река в 0.5 км: +1.2», «Снегозапас 142% нормы: +0.8»); модель объясняет и нули («Альжан: превышение 58 м → риск 0 — меры не требуются»). Клик по любой мере в очереди открывает «Почему рекомендовано»: сработавшее правило-порог, скор на момент генерации, разбор скора по факторам и формула приоритета. Где модель может ошибаться — честно описано в [Известных ограничениях](#известные-ограничения) (мало позитивов, PU-разметка, одно обучающее событие) — поэтому финальное решение всегда за комиссией.

## Быстрый старт (одна команда, нужен только Docker)

Работает одинаково на Windows / macOS / Linux — `make` не требуется:

```bash
docker compose up --build -d
```

Весь стек в docker-контейнерах: PostgreSQL (+PostGIS) → API → веб-клиент (nginx). Миграции, справочники и **данные пилота (520 НП СКО, ML-скоры, 17 сезонов) загружаются автоматически** при первом старте — ничего доустанавливать и настраивать не нужно (`.env` опционален: у compose есть значения по умолчанию).

- **Веб-клиент:** http://localhost:5173 — вход `admin@icybeard.local` / `ChangeMe123!`
- **API:** http://localhost:5080 (`/api/health`); из веб-клиента `/api` проксирует nginx
- Остановить: `docker compose down`; полный сброс БД: `docker compose down -v`
- На macOS/Linux есть сокращения: `make up` / `make upd` / `make down` / `make clean` / `make logs`

**Опционально:** очаги NASA FIRMS на карте пожарной обстановки требуют бесплатный ключ (2 минуты: firms.modaps.eosdis.nasa.gov/api → Generate MAP_KEY → в `.env`: `FIRMS_MAP_KEY=...`). Без ключа всё остальное работает, страница честно сообщает, чего не хватает.

**Демо-сценарий (2–3 минуты):** войти → Дашборд (KPI и очередь мер по всем контурам с разбивкой по угрозам, зимняя ML-сводка, последние решения комиссии) → «Ситуационная карта» (все три контура на одной странице, табы сверху): 🌊 Паводки — клик по селу → скор и факторы «почему» → «Сгенерировать черновики мер» → клик по мере → диалог «Почему рекомендовано» (правило, факторы, приоритет) → утвердить (фиксируется кто/когда) → селект «Сезон», date picker спутниковых слоёв (снимок/осадки/снег за любую дату с 2000 г.) и график снегозапаса → 🔥 Пожары — очаги, ветер, переключатель «Индекс | ML-прогноз», очередь мер по районам → ❄️ Зима — индекс по районам с разбором и мерами, переключатель «ML-прогноз» (ранг района за сезон + факторы «почему», пункт «сегодня» зимой) → «Данные» — загрузка CSV со скорами: карта сезона пересчитывается на глазах. На каждой вкладке — гистограмма распределения объектов по уровням риска (сколько НП/районов в красной зоне), у пожаров — топ-5 районов с переходом на карту.

Для локальной разработки с hot-reload (нужны .NET SDK 10 и Node.js 20+): `make db` + `make backend` + `make frontend`. Детали — в [backend/README.md](backend/README.md) и [frontend/README.md](frontend/README.md).

## Стек

- **Frontend:** Vue 3 + PrimeVue (шаблон [Sakai](https://sakai.primevue.org/)); карта — Leaflet + анимация ветра (leaflet-velocity) + спутниковые слои NASA GIBS
- **Backend:** .NET 10 minimal API, JWT-аутентификация, роли Admin/Analyst, прокси NASA FIRMS
- **DB:** PostgreSQL + PostGIS, EF Core миграции + автосид данных пилота при первом старте
- **ML/Data:** Python-пайплайн в `ml/` и `scripts/` (LightGBM + TreeSHAP); живая погода — Open-Meteo без ключей

## Как данные попадают на карту

1. Скрипты (`scripts/`) собирают признаки: OSM (НП, реки), Copernicus DEM (рельеф), ERA5-Land (снегозапас/осадки, 2000–2026).
2. `ml/i6-flood-risk/train.py` обучает LightGBM на разметке-2024 и отдаёт скоры + TreeSHAP-факторы; `score_years.py` скорит каждый сезон.
3. Загрузка: `PUT /api/settlements/metrics` (роль Admin) — либо автосид из CSV при первом старте контейнера.
4. Фронтенд: `GET /api/settlements/metrics/flood-risk?metricKey=risk_score&period=YYYY` → точки на карте, факторы в панели «Почему».
5. Live-слои мимо БД: Open-Meteo (погода/ветер), NASA FIRMS через `/api/fire/hotspots`, тайлы NASA GIBS (дата слоёв — из date picker'а: по умолчанию следует сезону, для пожаров — «вчера»).
6. Страница «Данные» (`/data`, Admin): загрузка CSV со скорами паводков — мгновенный ингест в БД и пересчёт карты сезона; прочие датасеты — в реестр для офлайн-пайплайна (`make fire-today` и др.).

## Правила для команды (2 человека)

- Документы кладём **только в `docs/`**, в подходящую тематическую подпапку.
- Внутренние коды модулей в путях/данных: `flood-risk`, `fire-risk` (наружу — «платформа природных рисков»).
- Сырые датасеты — в `data/raw/` (не коммитим, в `docs/data/` описываем, откуда скачать).
- Работаем в ветках `feat/<модуль>-<что-делаем>`, мерджим в `main` через PR.

## Известные ограничения

- Границы (`frontend/public/geo/kz-regions.geojson` и `kz-districts.geojson`) — geoBoundaries 2017 г.: 16 областей (без Абайской, Жетысу, Улытау) и 174 района. Заменить синхронно с сидером регионов (`DbSeeder.cs`).
- Разметка затоплений-2024 — 17 позитивов из СМИ (PU-постановка, методика в `docs/data/flood-labels-methodology.md`); модель обучена на одном событии.
- Очаги FIRMS берутся по прямоугольнику — видны точки и за границей РК.
- Дневная ML-модель пожаров работает на уровне районов; более гранулярная модель по ячейкам сетки — план основной программы.
